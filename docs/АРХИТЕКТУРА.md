# Архитектура

## Общая идея

Архитектура проекта должна поддерживать сервисный бизнес, а не мешать ему.

На старте всё строится вокруг простого потока:

`сайт / email / outreach -> n8n -> CRM / Telegram -> AI-черновики -> human approval -> delivery`

## Production-слой

Текущий живой стек на `Hetzner`:

- `Caddy` — reverse proxy и `HTTPS`
- `n8n` — workflow и orchestration
- `OpenClaw` — внутренний AI-интерфейс
- `scanner` — отдельный сервис
- `redis` — вспомогательный runtime-компонент

## Внешние операционные сервисы

- `Google Sheets` — стартовая CRM
- `Google Drive` — хранение клиентских файлов
- `Gmail` — почта
- `Telegram` — уведомления и approval

## Логическая схема

### 1. Intake

- сайт
- форма
- email
- ручной outreach

### 2. Оркестрация

- `n8n` принимает событие
- пишет лид в CRM
- отправляет уведомление в `Telegram`
- создаёт задачу или черновик ответа

### 3. AI-слой

- `OpenClaw` помогает создавать черновики
- AI не должен сам бесконтрольно отправлять клиентские материалы
- все внешние действия проходят через человека

### 4. Delivery

- brief
- audit
- menu pack
- tech cards
- food cost sheets

## Архитектурные правила

- один источник правды для лида
- approval перед клиентскими отправками
- секреты только вне репозитория
- инфраструктура должна быть минимальной
- новый сервис добавляется только если он реально упрощает поток

## Важная техническая заметка

Текущий repo-local `docker-compose` — это ещё не полный production mirror.

Живой production уже шире, чем локальный каркас в репозитории. Это нужно будет синхронизировать отдельной задачей.
